# 04 LINE OA และ Notification Design

ไฟล์นี้สอนแนวคิดการออกแบบระบบแจ้งเตือน โดยเริ่มจาก mock LINE OA ในหน้าเว็บ แล้วค่อยต่อยอดเป็น production

## Notification ไม่ใช่แค่ส่งข้อความ

หลายคนคิดว่า notification คือการส่งข้อความออกไป แต่ใน product จริงต้องคิดมากกว่านั้น:

- ส่งเมื่อไหร่
- ส่งหาใคร
- ส่งข้อความอะไร
- ส่งบ่อยแค่ไหนถึงไม่รบกวน
- ถ้าส่งไม่สำเร็จต้องทำอย่างไร
- token และข้อมูลลูกค้าถูกเก็บอย่างปลอดภัยไหม

ดังนั้น mock LINE OA ใน frontend มีไว้สื่อสาร flow ไม่ใช่แทน production architecture ทั้งหมด

## Event ที่ควรทำให้เกิดข้อความ

ระบบ loyalty มี event สำคัญ เช่น:

```text
ลูกค้าซื้อเพิ่ม
ลูกค้าใกล้ครบ 10
ลูกค้าได้ reward
ลูกค้าเก็บสิทธิ์ไว้ก่อน
ลูกค้าแลกของแถมแล้ว
ร้านมีโปรโมชันใหม่
```

แต่ละ event ไม่ควรใช้ข้อความเดียวกันหมด เพราะความตั้งใจต่างกัน

ตัวอย่าง:

```text
ใกล้ครบ 10 = กระตุ้นให้กลับมาซื้อ
ได้ reward = สร้างความดีใจทันที
เก็บสิทธิ์ไว้ก่อน = ยืนยันว่าระบบจำสิทธิ์ให้แล้ว
```

## ทำไม production ต้องมี backend

LINE OA ต้องใช้ access token ในการส่งข้อความ

ห้ามใส่ token ไว้ใน frontend เพราะผู้ใช้สามารถเปิดดู source หรือ network request ได้

architecture ที่เหมาะกว่า:

```text
Angular frontend
→ backend API
→ LINE Messaging API
```

backend มีหน้าที่:

- เก็บ token ใน environment variables
- ตรวจสิทธิ์การเรียก API
- บันทึก notification log
- retry เมื่อส่งไม่สำเร็จ
- ป้องกัน spam หรือส่งซ้ำ

## Mock กับ Production ต่างกันยังไง

Mock ในโปรเจกต์นี้:

```text
แสดงรายการข้อความในหน้าเว็บเพื่อสื่อสาร concept
```

Production:

```text
ส่งข้อความจริงผ่าน LINE OA และเก็บประวัติการส่ง
```

การแยกคำว่า mock ให้ชัดทำให้ portfolio ดูเป็นมืออาชีพ เพราะคนดูรู้ว่าเรารู้ข้อจำกัดและแผนต่อยอด

## UX ของข้อความที่ดี

ข้อความที่ดีควร:

- สั้น
- บอกสถานะชัด
- มีชื่อร้านหรือบริบท
- บอก next action เมื่อจำเป็น
- ไม่ใช้ถ้อยคำกดดันเกินไป

ตัวอย่างแนวคิด:

```text
คุณ A สะสมแล้ว 8/10 ไม้ อีก 2 ไม้รับของแถมได้เลย
```

ข้อความนี้ดีเพราะบอกทั้ง current state และ goal ที่เหลือ

## สิ่งที่ควรเรียนจากไฟล์นี้

ระบบแจ้งเตือนที่ดีคือการออกแบบ communication flow ไม่ใช่แค่เรียก API

ก่อนต่อ API จริง ควรถามเสมอว่า:

```text
event ไหนควรส่ง
ข้อความไหนช่วยลูกค้า
ข้อความไหนอาจรบกวน
ข้อมูลไหนห้ามอยู่ใน frontend
```
