# 01 Product Thinking จากปัญหาหน้าร้านจริง

ไฟล์นี้สอนวิธีเปลี่ยนไอเดียเล็ก ๆ จากหน้าร้านให้กลายเป็น web product ที่อธิบายใน portfolio ได้ ไม่ใช่แค่เขียนเว็บให้มีปุ่มกด

## เริ่มจากปัญหา ไม่ใช่เริ่มจากหน้าจอ

โจทย์ของระบบนี้คือร้านมีโปรโมชัน:

```text
ซื้อครบ 10 ไม้ แถม 1
```

ถ้ามองแบบ programmer อย่างเดียว เราอาจรีบทำช่องกรอกตัวเลขกับปุ่มบันทึก แต่ถ้ามองแบบ product developer ต้องถามก่อนว่า:

- ใครใช้ระบบนี้
- ใช้ตอนเวลาไหน
- ความผิดพลาดที่มีโอกาสเกิดคืออะไร
- ลูกค้าควรเห็นอะไรเพื่อรู้สึกเชื่อใจ
- ระบบนี้ช่วยให้ร้านดูต่างจากร้านทั่วไปอย่างไร

คำถามพวกนี้สำคัญ เพราะ software ที่ดีไม่ได้แค่คำนวณถูก แต่ต้องเข้ากับบริบทที่คนใช้จริง

## User หลักของระบบมีมากกว่า 1 คน

โปรเจกต์นี้มีผู้ใช้หลัก 3 กลุ่ม:

- พนักงานหน้าร้าน: ต้องบันทึกยอดเร็ว กดผิดแล้วแก้ได้
- ลูกค้า: ต้องเห็นยอดสะสมและเลือกของแถมเองได้
- เจ้าของร้าน: ต้องอยากนำระบบไปใช้จริง เพราะช่วยเก็บข้อมูลและดึงลูกค้ากลับมา

เวลาออกแบบหน้าเว็บ จึงไม่ควรคิดว่า "ทำหน้าเดียวให้สวย" แต่ควรคิดว่าแต่ละส่วนตอบโจทย์ user ไหน

## MVP คืออะไร

`MVP` ย่อมาจาก `Minimum Viable Product`

แปลแบบใช้งานจริง:

```text
เวอร์ชันเล็กที่สุดที่พิสูจน์ flow สำคัญได้
```

สำหรับโปรเจกต์นี้ MVP ไม่จำเป็นต้องมี backend จริงตั้งแต่วันแรก แต่ควรพิสูจน์ได้ว่า:

- เลือกลูกค้าได้
- บันทึกยอดซื้อได้
- ครบ 10 ไม้แล้วเกิดสิทธิ์ของแถม
- ลูกค้าเลือกของแถมเองได้
- ถ้ายังไม่ใช้สิทธิ์ ระบบต้องเก็บสิทธิ์ไว้
- มี mock LINE OA ให้เห็นทิศทาง production

## Feature กับ Learning ต่างกันยังไง

Feature คือสิ่งที่ระบบทำได้ เช่น:

```text
เพิ่มยอด 3 ไม้
แลกข้าวเหนียว 1 ห่อ
ส่งข้อความ LINE OA
```

Learning คือสิ่งที่เราเรียนรู้จากการทำ feature นั้น เช่น:

```text
การออกแบบ state ที่ไม่ทำให้สิทธิ์ลูกค้าหาย
การแยก component ให้คนอ่านเข้าใจ flow
การทำ CI/CD เพื่อให้ demo เปิดบน iPad ได้จริง
```

ไฟล์ใน `docs/teach/` ควรเน้น learning เป็นหลัก เพื่อให้กลับมาอ่านแล้วรู้ว่าโปรเจกต์นี้สอนอะไร

## สิ่งที่ควรเขียนลง portfolio

เวลาเอาโปรเจกต์นี้ไปโชว์ใน ApoRaviz Profile ให้เล่าแบบ case study:

- ปัญหาจริงของร้าน
- วิธีแปลงกฎโปรโมชันเป็น business logic
- UX ที่ลดการกดผิดใน POS
- iPad display ที่ทำให้หน้าร้านดูน่าสนใจ
- mock LINE OA ที่แสดงแผนต่อยอด
- สิ่งที่ยังเป็น mock และสิ่งที่ต้องทำเพิ่มถ้า production จริง

แนวนี้ทำให้คนดูเห็นทั้ง product thinking, UX thinking, และ engineering thinking พร้อมกัน
