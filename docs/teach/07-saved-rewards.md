# 07 State Modeling สำหรับ Saved Rewards

ไฟล์นี้สอนการออกแบบ state ให้ไม่ทำให้สิทธิ์ลูกค้าหาย โดยใช้กรณี `เก็บสิทธิ์ไว้ก่อน` เป็นตัวอย่าง

## State Modeling คืออะไร

State modeling คือการตัดสินใจว่าเราจะเก็บสถานะของระบบเป็นตัวแปรอะไรบ้าง และแต่ละตัวแปลว่าอะไร

ถ้าตั้ง state ไม่ชัด จะเกิด bug ง่าย เช่น:

```text
ลูกค้าเลือกเก็บสิทธิ์ไว้ก่อน แล้วสิทธิ์หาย
```

bug นี้ไม่ได้เกิดจาก UI อย่างเดียว แต่มาจาก model ที่ไม่แยกสถานะให้ชัดพอ

## แยก pending กับ saved ทำไม

ระบบนี้แยก reward เป็น 2 กลุ่ม:

```text
pendingRewards = สิทธิ์ที่เพิ่งเกิดและพร้อมให้เลือกตอนนี้
savedRewards = สิทธิ์ที่ลูกค้าเลือกเก็บไว้ใช้ครั้งหน้า
```

เหตุผลที่ไม่รวมเป็นตัวเดียว:

- UI ต้องบอกได้ว่าสิทธิ์ไหนควรถามลูกค้าตอนนี้
- สิทธิ์ที่เก็บไว้ต้องแสดงต่อเนื่อง
- การแลกของแถมควร consume จากแหล่งที่ถูกต้อง
- test ง่ายขึ้น เพราะ state transition ชัด

## State Transition คืออะไร

state transition คือการเปลี่ยนจากสถานะหนึ่งไปอีกสถานะหนึ่ง

ตัวอย่าง:

```text
ซื้อครบ 10:
sticks 9 + sale 1
→ pendingRewards + 1
→ sticks 0
```

ลูกค้าเลือกเก็บไว้ก่อน:

```text
pendingRewards 1
savedRewards 0
→ pendingRewards 0
→ savedRewards 1
```

ลูกค้าใช้สิทธิ์ภายหลัง:

```text
pendingRewards 0
savedRewards 1
→ savedRewards 0
→ reward redeemed
```

## Invariant คืออะไร

Invariant คือเงื่อนไขที่ควรจริงเสมอในระบบ

ตัวอย่าง invariant:

- `sticks` ต้องอยู่ระหว่าง 0 ถึง 9
- `pendingRewards` ต้องไม่ติดลบ
- `savedRewards` ต้องไม่ติดลบ
- เมื่อ save later จำนวน reward รวมต้องไม่หาย

เขียนเป็นคำจำง่าย:

```text
ก่อนและหลัง action สำคัญ สิทธิ์ของลูกค้าต้องนับได้เท่าเดิมหรือเปลี่ยนตามกติกาเท่านั้น
```

## ทำไม UI ต้องโชว์ savedRewards

ถ้าระบบเก็บสิทธิ์ไว้แต่ไม่แสดง ลูกค้าอาจรู้สึกว่าสิทธิ์หาย

ดังนั้น UI ควรแสดง:

- สิทธิ์ที่พร้อมใช้ตอนนี้
- สิทธิ์ที่เก็บไว้
- action สำหรับใช้สิทธิ์ที่เก็บ

นี่เป็นเรื่อง trust design: ระบบต้องทำให้ลูกค้าเชื่อว่าข้อมูลของตัวเองยังอยู่

## สิ่งที่ควร test

ควร test transition เหล่านี้:

- ได้ pending reward เมื่อครบ 10
- save later ย้าย pending ไป saved
- redeem ใช้ pending ก่อนหรือ saved ตาม logic ที่กำหนด
- จำนวน reward ไม่ติดลบ
- undo แล้ว pending/saved กลับถูกต้อง

ถ้า state เกี่ยวกับสิทธิ์ลูกค้า อย่าเชื่อแค่การลองกดเอง ควรมี test กัน regression
